  • 请求和响应报文

    • 请求报文

      • method,URL,版本
        • GET 获取资源
        • HEAD 相当于不返回Body的GET
        • POST 传输实体主体
        • CONNECT 要求在与代理服务器通信时建立隧道
      • 请求首部
      • 一个空行

      • 版本,状态码,描述

        • 2XX 正常处理完毕
        • 3XX 重定向
        • 4XX 客户端错误状态码
        • 5XX 服务器错误状态码

      • 首部

      • 空行

      • Body

  • Session

    • 和Cookie的区别是,Cookie是这样 的

Set-Cookie: userid_cookie=123

HTTP的演进

  • HTTP1.0

    • 无状态
    • 每次请求都需要建立一个TCP连接,处理完成后立即断开TCP连接,
    • 它存在的问题
      • 无法复用连接(TCP的连接释放过程比较费事)
      • 队头阻塞
        • 由于规定下一个请求必须在前一个请求响应之前才能发送,假设前一个请求一直不到达,那么下一个请求就不发送,后面的请求就阻塞了

  • HTTP1.1

    • 长连接: 通过设置Keep-Alive字段保持HTTP连接不断开

      • 和TCP keepalive的区别
        • HTTP keepalive是用于复用http连接,来避免每个资源都要创建一个HTTP连接,然后关闭
        • TCP keepalive 侧重于保持客户端和服务端的连接,会发心跳包给另一方,如果收到的不是ACK,而是发RST或响应超时就释放当前连接.防止因为其它原因没有四次挥手时,一方断开,另一方却不知道,一直维护者一个socket的描述符

    • pipelining(流水线)

      • 客户端可以一次性发出多个请求(而不是1.0种必须等到上一个请求已经响应了再发送),但是服务端依然需要以先后顺序响应.比如,客户端先后请求html和css文件,响应时,即使css先就绪也不能先发出
      • 事实上,浏览器可以打开多个TCP的会话,实现真并行

    • 缓存处理

      • 强缓存,协商缓存,启发式缓存

  • HTTP2.0

    • 1.x的缺陷
      • 需要多个连接才能实现并发和缩短延迟
      • 不会压缩请求和响应首部
      • 不支持有效的资源优先级
    • HTTP2.0
      • 二进制分帧(报文分为Headers frame和Data frame) (这个分层帧相当于再应用层和传输层之间)
        • 通信过程中,只有一个TCP连接存在,承载任意数量的Stream
        • 帧是最小的通信单位
      • 多路复用(连接共享)–真并行
        • 能够再一个TCP上进行任意数量的HTTP请求
        • 流是已建立连接上的双向字节流
        • 消息是与逻辑消息对应的完整的一系列数据帧
        • 帧是HTTP2.0通信的最小单位,每个帧包含头部,至少也会标识当前所属的流
        • pipeline中,虽然client可以一次性发出多个请求,但是在这些请求全部都得到恢复之前,会有队头阻塞的现象,相比之下多路复用中客户端发送的请求和服务端给出的响应的顺序不受限制
      • 服务端推送
        • 一次性把相关的资源传给客户端
      • 首部压缩
        • 要求S,C同时维护和更新一个之前见过的首部字段表
        • Huffman压缩

  • HTTP3.0

    • 加入了一个基于UDP的QUIC协议,并使用在了HTTP/3上

      • 0-RTT: 缓存当前会话的上下文,下次恢复时,只要将之前的缓存传递给服务器,验证通过,就可以进行传输了

      • 多路复用

        • QUIC基于UDP,一个连接上的多个stream之间没有任何依赖

      • 移动端表现

        • TCP通过IP识别,而QUIC通过ID识别

      • 除了个别报文,QUIC的packet头都经过认证,body都经过加密

  • SSL握手协议

    1. client hello: 客户端发起会话
    2. server hello: 服务器产生第一次应答
    3. certificate: 服务端向客户端发送自己的证书,(如果用RSA,就再发一个公钥;如果用DH,就下一步再发)
    4. server key exchange: EC(椭圆曲线)diffie-hellan , 服务端发送自己
    5. server hello done 一个空的指令 表示结束
    6. client key exchange 4中获得服务端的公钥,根据DH生成一个新的密钥对.客户端把自己的公钥传给服务端
    7. change cipher spec(客户端): 客户端的一个提示,表示主密钥已经生成了!可以开始对称密码了
    8. encrypted handshake message(客户端) 从上一步开始,信息全是加密的了
    9. Change cipher Spec: 服务端也生成主密钥
    10. encrypted handshake message(服务端)

HTTP的长连接和WebSocket的长连接

  • HTTP的长连接
    • 保证在一次TCP连接中可以完成多个HTTP请求
    • 但是对每个请求仍要单独发header
    • keep-alive不会永久保持连接,他有一个保持时间,可以在不同的服务器软件中设定这个时间
    • 双方没有建立真正的连接会话,服务端可以在任何一次请求完成后关闭
  • WebSocket的长连接
    • 是一个全双工的连接,可由服务器主动发起
    • 长连接第一次TCP链路建立后,后续数据可以双方都进行发送,不需要发送请求头
    • 是真正的双工的长连接,两边都必须要维持住连接的状态
